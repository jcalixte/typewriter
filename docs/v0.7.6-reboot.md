A user-triggered clean restart: `:reboot` (or the `> reboot` palette action, or a
future software button wired to the same `Effect::Reboot`). It auto-saves, paints
a branded "restarting..." screen, and resets.

## Why

Rebooting with the **hardware RST button** freezes the panel on its last editor
frame through the whole pre-app boot slice (2nd-stage bootloader + ~0.74 s PSRAM
memtest + `epd.init()`) until the async splash repaints — the "nothing changes"
gap. No firmware runs in that window, so a hardware-button reboot **cannot** show
an offboarding screen; the boot splash is already the earliest possible feedback.
A *software* reboot can: the firmware paints before it calls `esp_restart()`.

So `:reboot` exists to make an intentional restart *look* intentional, and to give
a software button a clean code path. It does **not** replace the RST button; it
sits alongside it. (Shrinking the hardware-button gap is a separate lever — the
boot-time `sdkconfig.defaults` levers, SPIRAM_MEMTEST off etc.)

## Behaviour

```
:reboot ──▶ auto-save every NAMED dirty buffer ──▶ paint reboot screen ──▶ esp_restart()
              (active first, then parked)            (blocking full refresh)

  dirty UNNAMED scratch buffer ──▶ refuse: "unnamed buffer - name it first"
```

- **Auto-save, then reboot** (chosen over refuse-if-dirty): a physical button that
  silently no-ops is confusing, so a reboot saves rather than blocks. Every named
  dirty resident buffer is saved — the active one through `write_active` (so it
  formats exactly like `:w`), each parked one via the same evict-time
  `Effect::Save`. The `Save`s are queued **ahead** of `Effect::Reboot`, and effects
  are serviced in order, so the host flushes them to the card **before** the reset.
- **Unnamed scratch buffer** — the empty-path scratch buffer has nowhere to save
  (`:w` posts "no file name"), so a *dirty* one blocks the reboot with a notice
  rather than lose the text. Essentially unreachable in the running editor (it
  always boots from a real file), but guarded for safety.

## The screen

`Frame::reboot()` reuses the boot splash's badge — the exact same stroked circle +
`typoena` wordmark (`draw_badge`, shared with `Frame::splash`) — plus a centred
`restarting...` line below the circle. Painted with a **blocking full refresh**, so
it is on the bistable panel before `esp_restart()` and simply carries over the
whole reboot into the boot splash: the restart reads as one continuous motion, not
a freeze.

## Scope notes

- **No git dependency.** Unlike `:setup` (which needs a marker + radio hand-off and
  is gated on `feature = "git"`), `:reboot` just paints and resets, so it works in
  the light build too.
- **`:reboot` vs `:setup`.** `:setup` reboots *into the wizard* (writes a marker
  first) and refuses when dirty; `:reboot` restarts *plainly* and auto-saves.
